# 后台富文本编辑器图片上传设计

更新时间：2026-04-01

## 目标

后台运营在编辑 Tours / Blog 内容时，需要以尽量低的操作成本插入图片，并且满足以下约束：

- 不要求手动输入相对路径或图片 URL。
- 图片保存后可以在前台正常展示。
- 不把大体积图片二进制直接塞进数据库。
- 保持当前项目结构简单，可继续沿用现有后台上传接口。
- 后续可以平滑迁移到对象存储或 CDN，而不需要重写编辑器交互。

## 结论

当前项目采用的方案是：

- 编辑器内直接选择本地图片、粘贴图片或拖拽图片。
- 前端将图片上传到后台 `/api/admin/upload`。
- 后台将图片保存到文件存储，并返回公开可访问的图片 URL。
- 富文本内容中仅保存图片 URL，不保存图片二进制，也不保存 Base64。

这也是主流富文本编辑器的常见实现方向：

- CKEditor 5 常见方案是使用 upload adapter，把图片上传到服务端后返回 URL。
- TinyMCE 通过 `images_upload_url` 或 `images_upload_handler` 上传后自动替换图片地址。
- ProseMirror / Tiptap 的常见方案是异步上传，完成后把占位内容替换为最终图片 URL。

参考：

- https://ckeditor.com/docs/ckeditor5/latest/features/images/image-upload/simple-upload-adapter.html
- https://www.tiny.cloud/docs/tinymce/latest/upload-images/
- https://prosemirror.net/examples/upload/

## 为什么不用数据库直接存图片

不采用“数据库直接存图片二进制”的原因：

- 数据库容量膨胀快，备份、恢复、迁移成本明显上升。
- 查询和传输链路会承受不必要的大对象负担。
- 图片更适合放在文件存储或对象存储中，由 Web 层单独缓存和分发。
- 未来如需接入 CDN、压缩、多尺寸图，这类能力更适合建立在文件 URL 之上，而不是数据库 BLOB。

也不采用“直接把 Base64 存进正文”的原因：

- HTML 内容显著膨胀。
- 页面传输体积更大，影响加载速度。
- 重复图片无法复用缓存。
- 后续内容迁移和图片治理都更麻烦。

## 当前实现

### 前端编辑器

文件：`frontend/components/admin/Editor.tsx`

当前支持三种插图方式：

- 点击工具栏图片按钮，打开本地文件选择器。
- 在编辑器中直接粘贴图片。
- 将图片拖拽到编辑器中。

行为流程：

1. 前端接收到图片文件。
2. 调用 `uploadAdminImage(file)` 上传到后台。
3. 后台返回图片 URL，例如 `/uploads/20260401_xxxxxxxx.webp`。
4. 编辑器在当前光标位置插入图片。
5. 保存文章时，正文中保存的是带 `src` 的 HTML。

### 上传接口

文件：

- `frontend/lib/admin-upload.ts`
- `backend/api/handlers/upload.go`
- `backend/api/routers/upload.go`

接口职责：

- 接收后台管理员上传的图片。
- 校验扩展名和文件大小。
- 保存到服务器上传目录。
- 返回图片访问地址。

当前后端已经支持的点：

- 扩展名白名单校验。
- 最大体积限制。
- 统一生成文件名，避免冲突。
- 返回可直接展示的 URL。

## 为什么这个方案最适合当前项目

从使用体验看：

- 运营不需要理解相对路径规则。
- 插图操作接近常见 CMS，学习成本低。
- 粘贴和拖拽能明显减少编辑成本。

从设计合理性看：

- 内容层只保存引用，不保存大文件本体。
- 编辑器与底层存储解耦，后续可以切换到 OSS / S3 / Cloudinary。
- 现有后端上传接口已经存在，改造成本低。

从性能角度看：

- 页面内容本身更轻，不会被 Base64 大幅放大。
- 图片可单独走浏览器缓存、反向代理缓存、CDN。
- 后续可在上传阶段做压缩、转 WebP、限制尺寸。

## 后续建议

当前版本已经满足“方便插入并正常展示”的核心目标，但还有三项推荐增强：

### 1. 上传阶段增加图片优化

建议在服务端增加：

- 自动压缩大图。
- 自动转 WebP 或保留原图并生成 WebP。
- 限制最大分辨率，例如最长边不超过 2400px。

这样能显著减少前台图片加载压力。

### 2. 增加媒体元数据表

如果后续需要更强的媒体管理能力，可以增加 `media_assets` 表，仅保存元数据：

- `id`
- `url`
- `original_name`
- `mime_type`
- `size`
- `width`
- `height`
- `created_by`
- `created_at`

注意：

- 这张表存的是“图片信息”，不是图片二进制。
- 富文本正文里仍然保存图片 URL。

### 3. 对象存储迁移

如果未来图片量增大，建议迁移到对象存储：

- 阿里云 OSS
- AWS S3
- Cloudflare R2
- Vercel Blob
- Cloudinary

迁移原则：

- 维持上传接口返回 `{ url: string }` 这一协议不变。
- 编辑器层不感知底层存储变化。

## 当前模块边界

当前实现只解决“编辑器插图上传并插入内容”的问题，不包含：

- 图片库管理页面
- 已上传图片复用选择器
- 上传进度百分比展示
- 图片裁剪
- 图片占位符节点跟随复杂编辑变动

这些能力如后续需要，可以在现有方案之上继续迭代。

## 维护建议

- 正文中始终保存图片 URL，不回退到 Base64。
- 上传接口保持统一返回格式，避免前端编辑器与存储实现耦合。
- 上传目录建议纳入备份策略。
- 生产环境建议由 Nginx 或 CDN 直接分发 `/uploads` 资源。

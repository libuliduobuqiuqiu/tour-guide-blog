# TikTok 同步排查记录

日期：2026-03-30
分支：`main`

## 一、背景

当前首页 TikTok 区块的真实需求是：

- 自动同步 TikTok 主页下的视频列表
- 获取每条视频对应的封面图
- 下载封面图并保存到本地系统
- 首页轮播使用本地缓存图展示
- 点击轮播卡片时跳转到对应 TikTok 视频页

这意味着 TikTok 同步链路真正需要的核心数据只有两类：

- 视频页链接
- 视频封面图链接

并不要求下载 TikTok 视频本身，也不要求接入评论、点赞等额外元数据。

## 二、这次排查的目标

本次排查主要回答下面两个问题：

1. 当前公开主页抓取链路为什么失败
2. 是否还能保留“自动同步”而不改成手动录入

## 三、已经验证过的路径

### 1. 直接解析 TikTok 公开主页 HTML

已验证：

- 桌面页 `https://www.tiktok.com/@tourjanet`
- 移动页 `https://www.tiktok.com/@tourjanet`

结果：

- 页面首屏都能拿到账号基础信息
- 页面脚本里可以拿到 `webapp.user-detail.userInfo.user`
- `videoCount` 为 `33`
- 但是 `itemList` 为空数组

结论：

TikTok 当前首屏返回的是“账号信息存在，但帖子列表不再直接放进 HTML”。

这也是之前后端报：

- `tiktok public sync found no posts`

的直接原因。

### 2. 解析页面内嵌 JSON 的更多结构

已额外尝试：

- `__UNIVERSAL_DATA_FOR_REHYDRATION__`
- `SIGI_STATE`
- `__NEXT_DATA__`
- `itemModule`
- `ItemModule`
- `itemList`
- `seo.abtest.vidList`

结果：

- `seo.abtest.vidList` 只拿到极少量实验标识，不能还原真实帖子列表
- `itemModule` / `itemList` 在该账号页面里没有返回可用的帖子数据
- 仅靠 HTML 中的脚本数据，无法还原首页需要的完整视频封面列表

结论：

单纯依赖“抓 HTML + 解析脚本 JSON”的后端方案，在当前 TikTok 页面结构下已经不足以完成自动同步。

### 3. 直接请求旧公开接口

已验证：

- `/api/post/item_list/`
- `/api/user/detail/`

结果：

- 接口会返回 `200`
- 但响应体为空

结论：

旧的无签名公开接口入口已经不再提供可直接消费的数据。

### 4. 使用 headless browser 执行页面脚本

已验证：

- 使用 Playwright + Chromium 打开 TikTok 用户页
- 监听页面实际触发的网络请求

结果：

浏览器执行页面脚本后，TikTok 确实会发起：

- `/api/post/item_list/`

并且 URL 中带有动态参数：

- `X-Bogus`
- `X-Gnarly`
- `device_id`
- `verifyFp`
- `msToken`

这说明：

- TikTok 用户页的视频列表不是完全不可得
- 而是已经变成了“客户端运行时生成签名请求，再异步拿数据”的模式

### 5. 浏览器环境下继续读取帖子列表接口返回体

已进一步验证：

- 在 Playwright 页面中复用真实签名 URL 再次 `fetch`
- 尝试以更接近真人浏览器的方式访问

结果：

- 页面本身出现了滑块验证提示：`Drag the slider to fit the puzzle`
- `/api/post/item_list/` 虽然仍返回 `200`
- 但响应体长度仍为 `0`
- DOM 中也没有出现 `/video/` 链接

结论：

TikTok 当前已经把这条链路推进到“浏览器运行 + 风控校验 + 签名请求”模式。

即使我们已经进入 headless browser 阶段，仍然会被滑块验证阻断，导致无法稳定拿到视频列表。

## 四、这次排查得到的明确结论

### 结论 1：当前失败不是代码简单 bug

这次失败并不是单纯的字段名变了、正则失效了，或者 JSON 路径少兼容了一个分支。

根因是：

- TikTok 首屏已不再直接公开帖子列表
- 帖子列表改为浏览器运行后异步请求
- 异步请求依赖动态签名和风控状态
- 当前无登录公开抓取链路已经不稳定甚至不可用

### 结论 2：用户想要的目标本身是合理的

目标很清晰，而且工程上也完全合理：

- 自动拿到视频链接
- 自动拿到封面图
- 下载封面到本地
- 首页用本地图片轮播展示

真正的问题不在目标，而在 TikTok 当前对公开网页数据源的限制。

### 结论 3：无登录、纯公开页面自动同步目前不可稳定实现

在当前验证结果下，下面这条路不能作为稳定生产方案：

- 仅靠 TikTok 公开用户页
- 不带登录态
- 不依赖浏览器会话
- 完全自动拿到完整视频封面列表

原因不是“理论上绝对不可能”，而是：

- 当前实测已经触发滑块验证
- 帖子列表接口即使命中也返回空体
- 这意味着稳定性无法接受

## 五、仍然可行的自动同步方向

如果目标必须保留为“自动同步”，目前只剩下这些现实可行方案。

### 方案 A：服务端使用 TikTok 登录态做浏览器同步

实现思路：

- 服务端保存一个可用 TikTok 登录会话或 cookies
- 用 Playwright 打开用户页
- 让页面真实执行脚本
- 读取异步加载后的帖子列表
- 提取视频链接和封面图
- 下载到本地系统
- 写入当前 feed 缓存

优点：

- 仍然符合“自动同步”要求
- 不需要手工逐条录入视频
- 更接近真实用户浏览行为

缺点：

- 需要维护登录态
- 登录态可能过期
- 仍可能遇到风控、验证码、区域限制
- 运维复杂度明显高于当前纯 HTTP 抓取

### 方案 B：改走 TikTok 官方或授权数据源

优点：

- 稳定性和契约性更强

缺点：

- 接入成本高
- 当前项目场景下未必现实

### 方案 C：第三方付费抓取服务

优点：

- 可以把反爬和签名问题外包

缺点：

- 有外部依赖
- 存在服务稳定性、成本和合规性问题

## 六、当前建议

如果项目明确要求“必须自动同步 TikTok 视频封面”，建议优先采用：

- 服务端登录态 + Playwright 浏览器同步

这是目前最接近现有目标、同时最有机会真正落地的方案。

原因：

- 纯公开抓取链路已经被 TikTok 当前页面模式和风控机制明显削弱
- 继续只修 Go 里的 HTML/JSON 解析器，投入和收益已经不成比例
- 下一步应把精力转到“浏览器同步能力”而不是“继续猜公开 JSON 结构”

## 七、这次已经落地的保护性改动

虽然 TikTok 自动同步尚未恢复，但这轮已经做了两个保护性修复：

1. 同步失败时不再覆盖已有 TikTok 缓存
2. 错误信息改得更准确，明确指出当前问题是：
   TikTok 页面只返回账号信息，帖子列表依赖签名客户端请求

这两个改动的目的，是避免一次失败把首页已缓存的 TikTok 内容直接清空，并提升后台排障可读性。
